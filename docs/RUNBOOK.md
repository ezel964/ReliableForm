# ReliableForm — Runbook

Day-to-day operations: start/stop, health, failure drills, queue and webhook
surgery, schema migrations, where the logs are. All commands assume the
project root as cwd.

See also: [API.md](API.md) (API v1 + webhooks reference) ·
[SRE-GUIDE.md](SRE-GUIDE.md) (why this playground exists, overhead A/B

## Start / stop / status / logs

```bash
bash launch.sh                  # start everything (default), prints ✓/✗ health table
bash launch.sh stop             # stop app services; MySQL/Redis stay up
bash launch.sh restart          # stop + start
bash launch.sh status           # per-service PID + healthz/heartbeat snapshot
bash launch.sh logs web1        # tail -f one service's log (nginx tails error+access)
bash launch.sh kill web1        # chaos drill: kill one service's whole process tree
bash launch.sh --yes            # any command, non-interactive (auto-yes to prompts)
```

- `start` is idempotent — it skips healthy services and automatically sweeps
  orphaned listeners off a service's port before binding (see Common issues).
- `stop`/`kill` never touch MySQL or Redis (system services).
- `kill <name>` takes down the full process tree (php -S forks workers that
  survive their master otherwise); `bash launch.sh start` brings it back.
- `kill`/`logs` accept **every** service name — `web1 web2 status worker-pdf
  worker-email worker-webhook otel nginx`. The parser binds the target before
  the command, so `bash launch.sh kill status` reads `status` as the service,
  not as a second command.
- First-time setup (deps, DB, schema+seed+migrations, npm install):
  `bash setup.sh` (re-run any time — see Schema migrations below).

## Service map

| service | what | port | log (storage/logs/) | pidfile (storage/run/) |
|---|---|---|---|---|
| `nginx` | load balancer / front door | 8080 (`LB_PORT`) | `nginx-access.log`, `nginx-error.log` | `nginx.pid` (written by nginx itself) |
| `web1` | PHP web app (`php -S`, 4 workers) | 9001 (`WEB1_PORT`) | `web1.log` | `web1.pid` |
| `web2` | identical second web instance | 9002 (`WEB2_PORT`) | `web2.log` | `web2.pid` |
| `status` | Node status & analytics service | 9301 (`STATUS_PORT`) | `status.log` | `status.pid` |
| `worker-pdf` | PHP CLI loop, consumes `queue:pdf` | — | `worker-pdf.log` | `worker-pdf.pid` |
| `worker-email` | PHP CLI loop, consumes `queue:email` | — | `worker-email.log` | `worker-email.pid` |
| `worker-webhook` | PHP CLI loop, consumes `queue:webhook`, POSTs deliveries | — | `worker-webhook.log` | `worker-webhook.pid` |
| `otel` | OTLP shipper sidecar — runs only when `OTEL_EXPORTER_OTLP_ENDPOINT` is set | — | `otel.log` | `otel.pid` |

MySQL (:3306) and Redis (:6379) are system services; `launch.sh` offers to
start them but never stops them.

## Health surfaces

| surface | semantics |
|---|---|
| `GET :8080/healthz` (via LB → web tier) | JSON `{"ok","instance","checks":{"db","redis"},"time"}` — **200** when both DB and Redis answer, **503** otherwise. Same endpoint directly on :9001/:9002. |
| `GET :9301/healthz` | `{"ok":true,"instance":"status"}` — liveness only, always 200 while the process runs. |
| `GET :8080/status` | human dashboard (auto-refresh 5s): per-service health, queue depths, worker heartbeats, stats. |
| `GET :8080/api/status` | same as JSON: probes lb/web1/web2/mysql/redis, `LLEN` of `queue:pdf`/`queue:email`/`queue:webhook` (flags depth > 100), heartbeat ages of all three workers. ~2s timeout per probe. |
| `GET :8080/api/stats` | usage analytics (users/forms/submissions/pdf+email job status counts/top forms). 503 when MySQL is down. |
| `X-Served-By` response header | which instance answered (`web1`/`web2`/`status`) — quickest way to watch the LB. |
| `X-Trace-Id` response header | 32-hex trace id of this request (web + status responses). |
| `X-Request-Id` | generated by nginx (`$request_id`) and forwarded to upstreams as a request header; appears as `rid=` in the access log and `request_id`/`app.request_id` in app logs/spans. Not echoed to clients. |
| `heartbeat:<instance>` Redis keys | SETEX 15s by `worker-pdf`, `worker-email`, `worker-webhook`, `status`, `otel` every loop. **Absent key = instance down.** |

Workers have no HTTP port — their liveness signal is the heartbeat key:

```bash
redis-cli GET heartbeat:worker-pdf    # unix ts, or (nil) = down
```

## Failure drills

### Kill a web instance

```bash
bash launch.sh kill web1
```

Expected: zero user-visible errors. Nginx ejects web1 (`max_fails=3
fail_timeout=5s`) and retries the other upstream (`proxy_next_upstream error
timeout http_502 http_503`); every `X-Served-By` becomes `web2`; `/status`
flags web1 within ~5s. Recovery:

```bash
bash launch.sh start
```

### Kill a worker

```bash
bash launch.sh kill worker-pdf
```

Expected: `queue:pdf` depth grows (`/status` flags depth > 100),
`heartbeat:worker-pdf` expires within 15s → `/status` shows the worker down;
submission pages show the "still rendering" PDF state. Nothing is lost — jobs
wait in Redis. Recovery: `bash launch.sh start` (the backlog drains; watch
`redis-cli LLEN queue:pdf` fall).

### Kill the status service

```bash
bash launch.sh kill status
```

Expected: `/status` and `/api/*` return 502 from nginx; the form app itself is
untouched (LB `/healthz` routes to the web tier and stays 200). Recovery:
`bash launch.sh start`.

### Stop Redis

```bash
brew services stop redis            # macOS; Linux: sudo systemctl stop redis-server
```

Note: `redis-cli shutdown nosave` is only a *blip* on a brew install — the
launchd service has KeepAlive and restarts redis within seconds. Use
`brew services stop redis` for a sustained outage.

Expected degradation per service:

| service | behavior with Redis down |
|---|---|
| web1/web2 | `/healthz` → 503 (`checks.redis:false`); sessions/login/CSRF unavailable; public form pages still render (cache falls through to DB). Submissions still commit to MySQL — queue pushes fail, user gets a flash warning, `web.submission.redis_degraded` increments. API submissions behave the same (still 201; `api.submission.redis_degraded`). Both rate limiters fail open. |
| workers | survive: log `redis unavailable, retrying until it returns`, poll every 3s, reconnect automatically. Heartbeats vanish (so `/status` calls them down — expected). |
| status | `/api/status` shows redis + both queues + both worker probes failing; `/api/stats` still works minus `submissions_last_hour`. |
| nginx | unaffected. |

Recovery:

```bash
brew services start redis           # Linux: sudo systemctl start redis-server
```

Workers reconnect on their own. Jobs pushed while Redis was down were never
enqueued — those submissions keep `pending` pdf/email rows (rows are created
in the MySQL transaction; only the queue push was lost).

## Queue surgery

Inspect:

```bash
redis-cli LLEN queue:pdf
redis-cli LLEN queue:email
redis-cli LLEN queue:webhook
redis-cli LLEN queue:dead
redis-cli LRANGE queue:dead 0 -1     # full dead-letter contents (JSON jobs)
```

A job that failed 3 attempts lands on `queue:dead` with an added `"error"`
field, and its DB row (`pdf_jobs`/`emails`/`webhook_deliveries`) is marked
`failed`.

Replay a dead-lettered job:

```bash
redis-cli LPOP queue:dead
# → {"type":"pdf","submission_id":123,"attempt":3,"enqueued_at":"...","trace":{...},"error":"..."}
```

Fix the root cause, then re-push with the attempt counter reset (drop the
`error` key; `type` tells you which queue):

```bash
redis-cli RPUSH queue:pdf '{"type":"pdf","submission_id":123,"attempt":0,"enqueued_at":"2026-06-11T00:00:00+00:00"}'
```

Replay is safe by design:

- Workers are **idempotent** — a job whose row is already `done`/`sent` is a
  no-op (checked before any work; `submission_id` is UNIQUE in both tables).
- Workers only UPDATE the `pdf_jobs`/`emails` rows the web app created;
  re-running cannot duplicate rows.
- Output files are written via temp-file + `rename()` (atomic), so a re-run
  just replaces `storage/pdfs/submission-<id>.pdf` / `storage/mail/submission-<id>.eml`.

Double-pushing the same job is harmless for the same reason — at worst the
second delivery is the no-op.

## Webhook operations

Full payload/header/retry reference: [API.md](API.md) → Webhooks.

One `webhook_deliveries` row per submission on a webhook-enabled form
(`submission_id` is UNIQUE; created by the web app in the submit transaction,
workers only UPDATE it):

| status | meaning |
|---|---|
| `pending` | row created, job queued (or queue push lost — see below) |
| `delivering` | worker picked it up; an attempt is in flight (or died mid-flight) |
| `delivered` | got a 2xx — `response_code`, `duration_ms`, `attempts` recorded |
| `failed` | 3 attempts exhausted — job dead-lettered, `error` holds the last failure (≤500 chars) |

Inspect:

```bash
mysql -h 127.0.0.1 -u reliableform -preliableform reliableform \
  -e "SELECT id, submission_id, status, attempts, response_code, error
        FROM webhook_deliveries ORDER BY id DESC LIMIT 10;"
```

### Replay a failed delivery

Same recipe as the dead-letter replay above: fix the receiver, reset the row,
re-push a matching job with the attempt counter at 0 (the dead-lettered job
on `queue:dead` tells you the submission id):

```bash
mysql -h 127.0.0.1 -u reliableform -preliableform reliableform \
  -e "UPDATE webhook_deliveries SET status='pending', error=NULL WHERE submission_id=123;"
redis-cli RPUSH queue:webhook '{"type":"webhook","submission_id":123,"attempt":0,"enqueued_at":"2026-06-11T00:00:00+00:00"}'
```

Safe by design: a row already `delivered` makes any re-pushed job a no-op,
and `attempts` keeps accumulating across replays (audit trail). The receiver
may still see the same delivery id twice — webhooks are at-least-once;
receivers dedup on `X-ReliableForm-Delivery`.

### Sink drills (deterministic failure testing)

```bash
php -S 127.0.0.1:9444 tools/webhook-sink.php   # logs one line per delivery
```

- Webhook → `http://127.0.0.1:9444/hook` — happy path, instant `delivered`.
- `...?fail=1` — every attempt 500s: watch 3 attempts in the sink log
  (backoff 2s, 4s), then the job on `queue:dead` and the row `failed`.
- `...?fail_first=2` — first 2 attempts of each delivery 500, then 200:
  "retry then deliver" — row ends `delivered` with `attempts=3`. Per-delivery
  state lives in `/tmp/rf-sink-<delivery id>.count`; remove those files to
  reset the drill.

Don't point a webhook at the stack's own LB/web ports (8080/9001/9002) unless
that loop is the drill — the worker logs a `self-loop drill hazard` warning
and delivers anyway.

## Schema migrations

`db/schema.sql` is always the **complete current schema** (fresh installs run
only it). Changes to existing databases ship as `db/migrations/NNN_name.sql`
and are applied by `setup.sh`, tracked in the `schema_migrations` table.

Adding a migration:

1. Create `db/migrations/NNN_short_name.sql` — next number, lexicographic
   order is execution order (e.g. `002_add_foo.sql`).
2. Apply the **same change** to `db/schema.sql` (it must stay complete).
3. Re-run `bash setup.sh`:
   - **fresh database** ⇒ schema.sql + seed.sql run, then every migration id
     is baseline-inserted into `schema_migrations` *without executing the
     file* (schema.sql already contains it);
   - **existing database** ⇒ each pending migration runs (as the app user,
     over TCP) and is recorded; already-recorded ids are skipped. A failing
     migration aborts setup before later ones run and records nothing.

Check state:

```bash
mysql -h 127.0.0.1 -u reliableform -preliableform reliableform \
  -e "SELECT * FROM schema_migrations ORDER BY id;"
```

Never edit an applied migration — ship a new one. If a migration ran but
recording failed, setup says so and prints the exact `INSERT` to run by hand.

## Frontend bundle & client telemetry

Plan 1 added a `frontend/` pnpm workspace (Rspack + TypeScript) that builds
hashed JS bundles served by nginx. PHP pages already inject the telemetry
bundle when it exists; the React dashboard/builder SPAs are not wired yet.

Build the injectable telemetry bundle (requires **pnpm**):

```bash
cd frontend && pnpm install && pnpm --filter @rf/telemetry-entry build
```

Outputs:

- `frontend/build/telemetry.<hash>.js` — RUM, trace propagation, JS error reporting
- `frontend/build/asset-manifest.json` — revision-hashed paths read by
  `apps/web/src/views/layout.php` (and future `FrontendLoader` SPAs)

Nginx serves bundles at `/build/` from disk (`config/nginx.conf.template` →
`location /build/` under `frontend/`). After changing the nginx template,
re-render and reload:

```bash
bash launch.sh restart    # re-renders storage/run/nginx.conf from the template
```

Verify: `curl -I http://localhost:8080/build/telemetry.<hash>.js` → **200**
with long-lived cache headers.

Frontend tests:

```bash
cd frontend && pnpm -r test
```

Backend beacon tests: `bash tests/run-unit.sh` (includes `test_rum.php`,
`test_clientlog.php`, `test_frontend_loader.php`).

### Telemetry env knobs (`.env`)

| variable | default | effect |
|---|---|---|
| `RUM_SAMPLE_RATE` | `1.0` | fraction of `/v1/rum` beacons that emit StatsD timings; `0` disables sampling |
| `CLIENTLOG_ENABLED` | `1` | set `0` to accept but discard all `/v1/clientlog` beacons (still 204) |
| `CLIENTLOG_RATE_PER_MIN` | `120` | per-IP cap on clientlog accepts; over-limit drops bump `web.client.clientlog_ratelimited` (Redis fail-open) |

To disable client telemetry entirely for a drill: `CLIENTLOG_ENABLED=0` and
`RUM_SAMPLE_RATE=0`, then restart web instances (`bash launch.sh restart`).

Metrics, AppLog `client_error` events, and frontend SLO targets:
[METRICS.md](METRICS.md) · [SRE-GUIDE.md](SRE-GUIDE.md).

## Uploads

Public-form file uploads land in `storage/uploads/<form_public_id>/` with
server-generated names (`<utcstamp>-<rand8>.<ext>` — never the client
filename). They are **write-only**: nothing ever serves them over HTTP, and
nothing reads them back (the submission stores only the path string). Old
test files are safe to purge at any time:

```bash
find storage/uploads -type f -mtime +7 -delete
```

## Log locations

All runtime logs live in `storage/logs/`:

| file | contents |
|---|---|
| `nginx-access.log` | one line per LB request, `sre` format (fields below) |
| `nginx-error.log` | nginx errors (upstream failures, config problems) |
| `web1.log`, `web2.log` | `php -S` stderr: built-in server request lines, PHP warnings |
| `status.log` | Node status service stdout/stderr |
| `worker-pdf.log`, `worker-email.log`, `worker-webhook.log` | worker lifecycle + per-job failure lines (webhook: also self-loop warnings) |
| `otel.log` | OTLP shipper sidecar output (exists only when the otel service runs) |
| `php-error.log` | PHP errors + uncaught exceptions from **all** PHP processes (web + workers), tagged `[instance]` |
| `app-<instance>.jsonl` | structured events: one JSON line per web request (`http_request`) / worker job (`job`) — field reference in [METRICS.md](METRICS.md) |
| `loadgen-runs.log` | loadgen A/B ledger, one pipe-separated line per run |

nginx `sre` log format:

```
$remote_addr [$time_local] "$request" $status $body_bytes_sent rt=$request_time upstream=$upstream_addr urt=$upstream_response_time rid=$request_id
```

- `rt=` total server-side time (seconds) — what loadgen computes percentiles from
- `urt=` upstream (web/status) response time; gap vs `rt=` ≈ nginx/queueing
- `upstream=` which backend served it (LB debugging)
- `rid=` request id; joins this line to `request_id` in `app-*.jsonl` and `app.request_id` on spans

## Common issues

| symptom | cause / fix |
|---|---|
| "Address already in use" on a service port | orphaned `php -S` children from an unclean death — `launch.sh start` sweeps them automatically (lsof-based) before binding; just re-run `bash launch.sh start` |
| MySQL admin login fails during setup on Debian/Ubuntu | apt installs auth root via `auth_socket` — setup retries with `sudo mysql` automatically; manual equivalent: `sudo mysql < db/schema.sql` |
| `launch.sh` says "app database/user is missing" | run `bash setup.sh` (creates DB + user, applies schema.sql + seed.sql, both idempotent) |
| status page shows "stats unavailable" | `apps/status/node_modules` missing — the service runs degraded without mysql2. Fix: `(cd apps/status && npm install)` or re-run `bash setup.sh` |
| files piling up in `storage/run/otel-spool/` | collector at `OTEL_EXPORTER_OTLP_ENDPOINT` unreachable — the shipper retries `.shipping.*` files and **drops them after 120s** (counts `otel.spans_dropped`). Fix the collector; nothing else to clean. |
| worker shows "alive but no heartbeat" | Redis down or the worker is stuck in a job — check `storage/logs/worker-*.log`, then `bash launch.sh kill worker-pdf && bash launch.sh start` |
| submissions saved but no PDF/email ever appears | jobs were lost while Redis was down (rows stay `pending`). Re-push by hand (see Queue surgery) using the submission id. |
| webhook delivery stuck `pending` forever | same cause: the queue push was lost (or `worker-webhook` is down — check `/status`). Re-push per Webhook operations → Replay. |
| no RUM/JS errors in metrics; pages lack telemetry script | frontend bundle not built — run the build in Frontend bundle & client telemetry; layout only injects when `frontend/build/asset-manifest.json` exists |
| `/build/...` returns 404 | bundle missing or nginx config stale — build the frontend bundle, then `bash launch.sh restart` to re-render nginx |
