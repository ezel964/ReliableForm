Why durable-first

One transaction writes the submission + all pending job rows; enqueue happens only after commit — a worker never sees a job before its row exists.

Graceful degradation

If post-commit Redis work fails, the submission still succeeds — the user never eats an infra failure.

Async pipeline

Workers · retries · dead-letters · driver fallback

  • Long-running PHP CLI loops — pdf / email / webhook.
  • Idempotent — re-running a done job is a no-op.
  • Retry with backoff — 3 attempts, then dead-letter.
  • Dead-letter — queue:dead, always inspectable.
  • Graceful shutdown — SIGTERM finishes the job, exits 0.
queue:pdf → rf_pdf queue:email → rf_email queue:webhook → rf_webhook

gearmand ↔ redis fallback

Kill gearmand → submissions still commit; pushes fall back to Redis lists, and workers drain them until gearmand returns.

Nothing stuck, nothing lost — no user-facing degradation.

Tracing

W3C trace context — end to end, no heavy SDK

We use the standard traceparent header so one submission can be followed across every process — web, workers, and the outbound webhook — with our own small lib/Trace.php, no vendor agent.

How it's wired

  • Incoming traceparent adopted on ingress (else a fresh trace starts).
  • Context embedded in every queue job → workers resume the same trace.
  • Forwarded on the outbound webhook — the trace leaves the stack.
  • Every response carries an X-Trace-Id header.

How it ships

  • Spans are written to a local spool file — never sent inline.
  • An otel sidecar ships them to any OTLP collector.
  • One switch: OTEL_EXPORTER_OTLP_ENDPOINT in .env.
  • Unset ⇒ tracing fully off, zero cost on the request path.
JotForm propagates its own JF-Trace-Parent-ID; we use the open W3C standard — same idea, portable to any tracing backend.
